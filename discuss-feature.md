# Discuss Feature Implementation

Gather context and implementation decisions BEFORE creating tasks. This ensures task creation has clear direction on gray areas and user preferences.

Run this before `/ccmagic:create-tasks` to produce a `CONTEXT.md` that guides task generation.

## Process

### 1. Load Feature

If `$ARGUMENTS` provided, use that feature path. Otherwise, list features and ask user to select.

Read the feature's `overview.md` from `context/features/[feature-path]/`.

### 2. Analyze for Gray Areas

Review the feature overview and identify:

**Technical Decisions:**
- Architecture choices not yet specified
- Library/framework options
- Data model questions
- API design decisions
- Performance vs simplicity tradeoffs

**Scope Boundaries:**
- What's explicitly in scope
- What might be out of scope
- Edge cases that need clarification
- MVP vs future considerations

**Integration Points:**
- How this connects to existing code
- External dependencies
- Breaking changes potential

### 3. Adaptive Questioning

Use AskUserQuestion to clarify each gray area discovered. Group related questions.

Example question categories:

**Architecture:**
```
How should [X] be implemented?
- Option A: [approach] - [tradeoff]
- Option B: [approach] - [tradeoff]
- Option C: Let me decide based on codebase patterns
```

**Scope:**
```
Should [edge case] be handled in this feature?
- Yes, it's required for MVP
- No, add to backlog for later
- Depends - let's discuss
```

**Conventions:**
```
For [decision point], should we:
- Follow existing pattern from [file/module]
- Introduce new pattern because [reason]
- Research best practices first
```

### 4. Explore Codebase (if needed)

If user selects "follow existing patterns" or "research", use Explore agent:

```
Use Task tool with:
  subagent_type: "Explore"
  prompt: "Find existing patterns for [topic] in this codebase.
          Look for: [specific patterns needed]
          Report: file locations, approach used, any inconsistencies"
```

### 5. Generate CONTEXT.md

Write to `context/features/[feature-path]/CONTEXT.md`:

```markdown
# Feature Context: [feature-name]
Generated: [timestamp]
Feature: [feature-path]

## Implementation Decisions

### [Decision Category 1]
**Question:** [What was asked]
**Decision:** [User's choice]
**Rationale:** [Why, if provided]

### [Decision Category 2]
...

## Scope Boundaries

### In Scope
- [Item explicitly included]
- [Item explicitly included]

### Out of Scope (Backlog)
- [Item deferred] → Add to backlog after feature complete
- [Item deferred]

### Edge Cases
- [Edge case]: [How to handle]

## Technical Approach

### Architecture
[Summary of architectural decisions]

### Patterns to Follow
- [Pattern]: See [file] for reference
- [Pattern]: See [file] for reference

### Dependencies
- [Dependency]: [How it will be used]

## Constraints

- [Any constraints identified]
- [Performance requirements]
- [Compatibility requirements]

## Open Questions (for task implementer)

- [Any remaining questions that can be resolved during implementation]

---
*This context was generated by `/ccmagic:discuss-feature` and should be read by `/ccmagic:create-tasks`*
```

### 6. Prompt Next Step

After writing CONTEXT.md:

```
Feature context captured in CONTEXT.md.

Next: Run `/ccmagic:create-tasks [feature-path]` to generate tasks with this context.
```

## Notes

- Run this BEFORE `/ccmagic:create-tasks`, not after
- Questions are adaptive - fewer for simple features, more for complex ones
- If user is unsure, capture as "Open Question" for implementer
- CONTEXT.md becomes part of the feature's permanent record
- Task creation should read CONTEXT.md to inform task breakdown
